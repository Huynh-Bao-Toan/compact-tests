# Naming conventions

## Feature/module naming

* Feature name = the controller name **without** the `Controller` suffix
  (`WidgetsController` -> `Widgets`), unless project evidence suggests a different grouping (e.g. two
  controllers that both belong to one business module/screen). If you deviate from the default, state
  why in your coverage report.
* Set `Feature` explicitly on every `CompatibilityCase` - don't rely on `CompatibilityCase`'s
  `Name`/`SnapshotName` derivation fallback for irregular plurals or multi-word names (see the
  `NOTE` comment on `CompatibilityModels.cs`'s `DeriveFeatureAndKey()` - it's a convenience for
  regular-shaped names only).

## Case file naming

* One controller/module = one cases file: `Compat.Legacy/Cases/<Feature>Cases.cs`
  (e.g. `WidgetsCases.cs`).
* Start from `Compat.Legacy/Cases/ExampleCases.template.cs` (copy, rename, fill in) rather than
  writing a cases file from scratch - it already has the right shape and file-exclusion setup.
* Mirror the same feature's cases into `Compat.Migrated/MigratedVerificationTests.cs`'s `Cases()`
  when the migrated side is ready to verify (see `Compat.Migrated/Cases/README.md`).

## Snapshot file naming

* One controller/module = one grouped snapshot file: `snapshots/<Feature>.approved.json`.
* Never hand-write this file - it's generated by running legacy snapshot generation (see the root
  `README.md`) and reviewed via its git diff.
* On a migrated-verification mismatch, a sibling `snapshots/<Feature>.received.json` is written for
  inspection - this file is diagnostic output, not something to commit (see the root `.gitignore`).

## Case key naming

* Format: `<Action>/<scenario_key>` (e.g. `AddProducts/number_as_string`).
* Build it with `CompatibilityCaseKey.For("<Action>", CompatibilityScenarios.X)` rather than
  string-concatenating it inline - this guarantees the `/` separator and rejects empty
  action/scenario at the point of authoring instead of producing a malformed key silently.
* `<Action>` is the literal MVC action method name (as it appears in the controller), not the JS
  function/service method name that calls it, if they differ.

## Scenario key naming

* `lowercase_snake_case`.
* Describes the **input shape/condition** being tested, not the expected outcome:
  good: `number_as_string`; avoid: `number_as_string_should_coerce_to_int`.
* One scenario key = one distinct risk. Don't fold two unrelated risks into one scenario
  (e.g. don't invent `number_as_string_and_missing_field` - author two cases with two scenario keys).

## Global vs. feature-specific scenarios

| Kind | Lives in | Use when |
|---|---|---|
| **Global** | `Compat.Core/Scenarios/CompatibilityScenarios.cs` | The risk is a generic MVC5 -> ASP.NET Core binding/serialization concern that could recur on any controller (type coercion, empty/missing values, request encoding, casing, collection null vs empty, date/decimal formatting, duplicate query keys). |
| **Feature-specific** | `Compat.Legacy/Cases/<Feature>/<Feature>Scenarios.cs` (create only when needed) | The risk is a business rule specific to one controller/module - not something another controller would ever hit. |

Rules:

1. **Check `CompatibilityScenarios` before inventing a new scenario key**, global or feature-specific.
   Reusing an existing constant (even if the wording isn't a perfect match) beats creating a
   near-duplicate - near-duplicates are exactly what this taxonomy exists to prevent.
2. **Do not create a feature-specific scenario for a risk already covered globally.** If a case is
   "the frontend sent shape X, does the migrated binder still accept it," it's a global scenario
   candidate even if the field names are feature-specific - the field names go in the case's `Body`,
   not the scenario key.
3. **Do not add a new global scenario for a one-off business rule.** If only one controller could ever
   hit this risk, it belongs in that feature's own `<Feature>Scenarios.cs`, not
   `Compat.Core/Scenarios/CompatibilityScenarios.cs`.
4. When you do add a new global scenario, add one `public const string` to `CompatibilityScenarios`,
   grouped alphabetically within its existing category comment block, and use it via
   `CompatibilityCaseKey.For(action, CompatibilityScenarios.YourNewOne)`.
